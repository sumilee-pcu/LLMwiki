# AI 답변 물타기 방지 모드

## 요약

AI 답변 물타기 방지 모드는 LLM이 지나치게 친절한 리액션, 질문 재포장, 과도한 헤더와 목록, 안전한 양비론, 근거 없는 확신으로 답변 품질을 희석하는 문제를 줄이기 위한 커스텀 인스트럭션 패턴이다.

목표는 차갑게 말하는 것이 아니라, 정보 밀도와 판단 품질을 높이는 것이다. LLMwiki에서는 코드 리뷰, 연구 메모, 팩트체크, 도구 추천, 정책 비교처럼 정확성과 결론이 중요한 작업에 우선 적용한다.

## 핵심 원칙

| 원칙 | 설정 | 기대 효과 |
| --- | --- | --- |
| 불필요한 리액션 제거 | "Great question", "Absolutely" 같은 시작 문구 금지 | 바로 본론으로 들어가 답변 밀도 증가 |
| 과도한 포맷팅 제한 | 기본은 자연스러운 문단, 필요한 경우에만 목록·표 사용 | 덜 기계적인 글, 읽는 피로도 감소 |
| 정직 모드 | 문제 있으면 먼저 직접 지적, 칭찬-비판-칭찬 구조 금지 | 실질 피드백 증가 |
| 가정 공개 | 전제, 해석, 트레이드오프를 먼저 밝힘 | 오해와 임의 해석 감소 |
| 반복 제거 | 질문 재진술, 장황한 재포장 금지 | 속도감과 신뢰감 증가 |
| 확신 수준 표시 | 사실·추론·추측을 구분하고 불확실성 명시 | 환각 체감 감소 |
| 의견 요청 시 선택 | A/B 질문이면 하나를 고르고 이유를 설명 | 의사결정 지원 강화 |

## 커스텀 인스트럭션 예시

아래 문구는 ChatGPT, Claude, Gemini, Grok 등에서 사용자 성향에 맞게 줄여 쓸 수 있다.

```text
답변은 정보 전달 중심으로 작성한다.

불필요한 리액션 문구로 시작하지 않는다. 예: "좋은 질문입니다", "물론입니다", "흥미로운 지점입니다" 같은 표현을 쓰지 않는다.

사용자의 질문을 길게 재진술하지 않는다. 바로 답한다.

모르면 모른다고 말한다. 불확실한 내용은 사실, 추론, 추측으로 구분한다.

사용자의 전제가 틀렸거나 위험하면 먼저 지적한다. 칭찬-비판-칭찬 구조로 완화하지 않는다.

기본은 짧은 문단으로 답한다. 목록, 표, 헤더는 비교·절차·체크리스트가 필요할 때만 쓴다.

A와 B 중 선택을 요청받으면 하나를 고르고 이유를 설명한다. 다만 조건에 따라 결론이 달라지면 그 조건을 먼저 밝힌다.

핵심 전달이 끝나면 불필요한 마무리 문장을 덧붙이지 않는다.
```

## 적용이 좋은 상황

| 상황 | 이유 |
| --- | --- |
| 코드 리뷰 | 문제를 먼저 지적해야 수정 비용이 줄어듦 |
| 팩트체크 | 사실·추론·추측 구분이 중요 |
| 도구 추천 | 애매한 중립보다 선택과 근거가 필요 |
| 정책·가격 비교 | 시점, 조건, 불확실성 표시가 핵심 |
| 논문·기획 피드백 | 약점과 전제 오류를 빨리 드러내야 함 |
| 에이전트 운영 규칙 | 반복·장식 문장을 줄이면 context window 효율 증가 |

## 완화가 필요한 상황

모든 대화에 강하게 적용하면 답변이 딱딱해질 수 있다. 다음 상황에서는 톤을 조금 부드럽게 둔다.

| 상황 | 조정 |
| --- | --- |
| 초보자 교육 | 직접 지적은 하되, 다음 행동을 함께 제시 |
| 학생 상담 | 정보 밀도보다 정서적 안정과 안전 우선 |
| 창작 피드백 | 문제점과 가능성을 함께 제시 |
| 브레인스토밍 | 하나를 고르기 전 여러 가능성을 잠깐 펼쳐도 됨 |
| 고위험 의사결정 | 하나를 선택하더라도 불확실성과 전문가 확인 필요성을 명시 |

## LLMwiki 적용

LLMwiki에서는 이 모드를 다음 운영 규칙으로 해석한다.

| LLMwiki 작업 | 적용 규칙 |
| --- | --- |
| 출처 카드 작성 | 확인된 사실과 해석을 분리 |
| 개념 페이지 작성 | 요약을 먼저 쓰고, 장식 문장 제거 |
| GitHub 저장소 평가 | 장점보다 리스크와 적용 조건을 먼저 확인 |
| 코드·도구 검증 | "작동 가능성"과 "실제 검증 여부"를 구분 |
| 에이전트 설계 | context window에 들어갈 문장을 짧고 명확하게 유지 |

이 패턴은 [[12-Factor Agents]]의 "Own your context window", [[SkillOpt]]의 skill 문서 최적화, [[Harness Engineering|하네스 엔지니어링]]의 검증 루프와 직접 연결된다.
